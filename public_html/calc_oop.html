    How the calculator program demonstrates OOP Principals

    Encapsulation:

    Picture by Bob Schicke
    Encapsulation using a class encloses, or wraps variables and methods in s group whose members have protected access. In a class, external access is only allowed to methods or variables that are designated as public. Generally the data is private (non-excessible) except by using special public functions frequently called setters and getters. The same is true for functions that deal with critical information such as websites and databases. You want to make the critical methods private and use public functions to form a safe barrier. In our Calculator project we use encapsulation in all of our classes. For example, our Calculator class encapsulates the add, subtract, multiply and divide methods.

    Inheritance:

    Picture by Bob Schicke
    Inheritance allows you to have a general parent class that is inherited by the child classes. This can make sense if the child classes can use most of the methods and variables from the parent. For instance, a parent class can be “Animals” and have variables that would apply to all animals such as weight, color, height, length, life expectancy, habitat. And some children of animals would be Mammals, Reptiles, Fish, Birds, and these child classes would add data and methods that was specific to their class, such as fin size, wing span, hair length. In our Calculator project we have a parent class Calculation that has a tuple “values” and a method to convert “values” to a list of floats. We then have Addition, Subtraction, Multiply and Divide classes that inherit Calculation so that they can use Calculation's members. After inheriting the from Calculation, the children only need to implement a “get_result” function, which needs to be different for each of the operations performed by the children.

    Polymorphism:

    Picture by Bob Schicke
    Polymorphism means that you have the same function name, but have custom implementations based on the type of object or type of data. One example of this in python is the “print” method. For example, the print method has custom implementations for integers, strings, lists. You don't need to be concerned with letting the function know what type of data is being printed. The correct implementation is called based on the data that is passed. You can create your own polymorphism by having a property or method in a parent function and then creating a custom implementation in some of the child classes. We use polymorphism in the Calculator project when the Addition, Subtraction, Multiplication and Division classes implement custom “get_result” methods. When get_result(values) is called on any of the classes it returns the value of the operation.

    Abstraction:

    Picture by Bob Schicke
    Abstraction is the process using a simple interface, such as function calls, to hide the complexity of the underlying code. When you type in a function call, such as the print(“Hello World”) function in python, there may be 100+ lines of code to correctly implement the function. A similar thing is when you type a phone number into your phone, enter a search string in google or turn the key in your car. You do a very simple thing, and a very complex process occurs. In our Calculator project, there are many abstractions that take place. For example, when we call the get_result function, the program loops through the values and performs the operations and then returns the result. The code of the get_result function is hidden from the caller.

    And in another layer of abstraction, there is library code that implements the program code above. As programmers we don't necessarily care how a “for” loop is coded in the library. We just need to know how to use it in the program code we write. In the early days of programming, you had to program in machine code. Now we can write 10 lines of code in a few minutes that does more than a program that used to take a week to write on 1000 punch cards, all thanks to abstraction.

    JCR Licklider

    "JCR Licklider" by atduskgreg is licensed under CC BY-NC-ND 2.0
    Licklider was born on March 11, 1915, in St. Louis, Missouri. He studied at Washington University in St. Louis where he graduated with a triple major in physics, mathematics and psychology and went on to his doctorate. He became a research fellow at MIT in 1943 and in 1950 he became interested in information technology. He had ideas about graphical point and click interfaces, digital libraries, e-commerce and online banking. He learned about time-sharing at a conference in Paris, and developed his own version of time-sharing. In 1962 Licklider joined ARPA (Advanced Research Projects Agency) as head of the Information Processing Techniques Office. His work at ARPA included conceiving of and funding network research. While at IPTO, he convinced some of the most important players in the development of the internet, Ivan Sutherland, Bob Taylor, and Larry Roberts, that an all-encompassing computer network was very important.
