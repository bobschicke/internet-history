    AAA Pattern for Unit Testing

    What is Unit Testing?

    "Sputnik-1" by paukrus is licensed under CC BY-NC-ND 2.0
    Unit Testing is a way of testing each part (unit) of the program for proper functionality. Python has a module in its standard library called unittest that contains tools for testing your code. To implement unit testing, you put "test" methods in your project to call and test the "real" methods in your code. The AAA pattern is one way of implementing unit testing.

    What is AAA Pattern?

    "Mainframe Computer 1960" by PanelSwitchman is licensed under CC BY-NC-ND 2.0
    AAA stands for Arrange, Act and Assert. Arrange means to setup the test by creating objects, assigning data to variables, signing into websites or databases, etc. Act means to execute the test which can involve calling methods, passing data or interacting with a website. Assert means to check that the method under test performed the operations correctly. Let's see a simple example: 
                            # This is a function that adds 2 numbers
                        def add(val1, val2):
                            return val1+ val2

                        # This is our test function
                        def test_add():
                            #Arrange
                            value1 = 5
                            value2 = 6
                            #Act
                            result = add(value1, value2)
                            #Assert
                            assert result == 11
    In the above example the Arrange section instantiates variables and assigns values to them. The Act calls the function that we are testing and passes the variables to it and then assigns the return value to result. The Assert section checks that the value that was returned, the database values, website status, are what was expected. If they are do not match the expected results then the test will fail. It's important to test often, so you don't get too far down the wrong path. If you code a small section and test it, it will be much easier to debug than if you code for a week and then test it. You should test at least as often as you commit, and it's good to commit a couple times a day. If you follow these practices you will never get too far astray.

    How to Configure Against Nuisance Warnings

    "Network" by Qaanaaq is licensed under CC BY-NC-ND 2.0
    Pylint errors and warnings are very descriptive and point you to the file, line number and column number. It really couldn't get any easier. That said, sometimes pylint gives warnings for things you are doing on purpose. The answer to that is to add a directive to disable the warning. For example, if you want to disable the warning for a line that is too long, simply add this command before the offending code: # pylint: disable=line-too-long

    JCR Licklider

    "JCR Licklider" by atduskgreg is licensed under CC BY-NC-ND 2.0
    Licklider was born on March 11, 1915, in St. Louis, Missouri. He studied at Washington University in St. Louis where he graduated with a triple major in physics, mathematics and psychology and went on to his doctorate. He became a research fellow at MIT in 1943 and in 1950 he became interested in information technology. He had ideas about graphical point and click interfaces, digital libraries, e-commerce and online banking. He learned about time-sharing at a conference in Paris, and developed his own version of time-sharing. In 1962 Licklider joined ARPA (Advanced Research Projects Agency) as head of the Information Processing Techniques Office. His work at ARPA included conceiving of and funding network research. While at IPTO, he convinced some of the most important players in the development of the internet, Ivan Sutherland, Bob Taylor, and Larry Roberts, that an all-encompassing computer network was very important.
